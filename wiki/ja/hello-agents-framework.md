# HelloAgents フレームワーク

## 概要

HelloAgents は、Datawhale の *Hello Agents* チュートリアルで導入される軽量 AI エージェントフレームワークで、読者をゼロからのフレームワーク構築へ導きます。本番向けの重量級フレームワークではなく、**教育指向**のフレームワークであり、核心理念は「万物皆為工具（Everything is a Tool）」です。

## 設計思想

1. **軽量 + 教育フレンドリー**：コアコードは章ごとに整理され、依存関係は最小限（OpenAI SDK などの必須ライブラリのみ）で、可読性が高い。学習者は何万行ものフレームワークコードの中で迷うことなく、具体的なコード行まで追跡できます。

2. **標準 API ベース**：OpenAI API 仕様に完全準拠し、移行と統合のコストが低い。あらゆる OpenAI 互換サービス（ローカルモデルを含む）を接続可能です。

3. **段階的な学習パス**：フレームワークは pip バージョン（例: `hello-agents==0.1.1`）で反復し、各バージョンが学習章に対応。学習者は段階的に深く進めます。

4. **統一されたツール抽象化**：Memory、RAG、MCP などの複雑なモジュールをすべて「ツール」として統一抽象化。抽象化レイヤーを減らし、エージェントの本質 — ツールを呼び出してタスクを達成すること — に立ち返ります。

## アーキテクチャ階層

```
hello_agents/
├── core/          # コアフレームワーク層（Agent基底、LLMインターフェース、メッセージシステム、設定管理）
├── agents/        # エージェント実装層（Simple, ReAct, Reflection, PlanAndSolve）
└── tools/         # ツールシステム層（基底、レジストリ、非同期実行、組み込みツール）
```

3層分離により、各層を独立して拡張できます。新しい LLM プロバイダーを追加してもエージェントロジックに触れる必要はなく、新しいエージェントパラダイムを追加してもツールシステムを変更する必要はありません。

## なぜ自前のフレームワークを構築するのか？

主流フレームワーク（LangChain など）には次の問題点があります：

- **過剰な抽象化**：概念が多すぎて、単純なタスクでも大量の前提知識の理解が必要
- **API の不安定性**：バージョン反復が速く、メンテナンスコストが高い
- **ブラックボックスロジック**：コア実装が不透明で、深いカスタマイズが困難
- **依存関係の複雑さ**：パッケージサイズが大きく、依存関係の競合が発生しやすい

自前フレームワークの構築価値は：エージェントの動作原理の深い理解、完全な制御権の獲得、システム設計能力の育成にあります。特に金融、医療、教育などの垂直領域では、カスタマイズ要件が汎用フレームワークの提供できる柔軟性をはるかに超えることがよくあります。
